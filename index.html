What is

the “Collaborative Tool”?

a bit of context and areas of challenges

Tool basics

  1. Melbourne Water-led initiative to support the Healthy Waterways Strategy
  2. Web-based digital tool with data and information about what's happening in the catchment
  3. Designed to be used by all catchment stakeholders
  4. Has the (ambitious) goal of 'collaboration'
From the review and scoping study: "A key contention of this review is that the name ‘Collaborative Tool’ evokes very different meanings and expectations to different partners and stakeholders. These meanings are shaped by the characteristics and past experiences of each collaborating partner and are grounded in the relationship that they have with others who they are trying to collaborate with. Moreover, there are asymmetries in information, resources, capabilities and power between each partner that makes it extremely challenging - if not impossible - to design a truly ‘democratised’ collaboration Tool.""

Supports the Healthy Waterways strategy

The drivers of the Tool are linked to performance objectives of the HWS
  • RPO-44 (Pp. 165): Web-based systems are established to report performance and measure outcomes of the catchment implementation forums.
  • RPO-39 (Pp. 156): Systems and pathways to share knowledge and information. No delivery date specified.

Potential tool user types

The HWS has over 200 'stakeholder' organisations. Some are tool users. Some are key data providers. Some are both. Most have different needs and expectations.
Stakeholder mappping exercise taken from Collaborative Tool Scoping Report (2021)

Collaboration: the data platform dilemma

  • Need compelling content (incl. data/information) for users to use the Tool
  • Users might become content contributors (but we don't know in what capacity/form)
  • User contributions make the Tool content more compelling.

Why a Prototype → pilot?

  • Need to test ideas and get feedback (and re-design)
  • Need to use existing data and information (no 'data mock ups')
  • Makes sense to break it down into smaller 'pilot', then scale what works
  • Want to work (as) rapidly (as possible)...
...to 'build' something we can get real feedback on and make better
(akin to a 'lean startup' approach)

The process of

Design

Understanding users + developing technical architecture
Starting with two intertwined areas of

Design research

  • Understanding users: user research and testing (traditional 'design' or 'co-design'). We need to talk to potential users to understand their needs
  • Technical architecture: data and information review and 'engineering'. Reviewing existing data for quality, completeness and accessibility.
Designing a data driven user interface

Basic User flow + supporting data

Research task

Data review

  • Critical to get actual data: quality checking + need to know 'what it can do' and if there are gaps
  • Test Data pipelines: need to understand how that data is created, and how it can be accessed. And if the Tool's data can be usefully fed back*.
  • Understand data owners: need to establish relationships with data owners
*The prototype won't provide this but will be designed to later provide a data API
Research task

User interviews

  • Informal discussions: gauge general interest, assess capabilities, understand needs and pain points (that might be related to tool...or not)
  • Compatibility with internal processes and systems: understand where the Tool might fit in, and if there worthwhile tradeoffs to make in Tool design to encourage use
  • Feature wishlists: Encourage users to think about 'possibilities' of the Tool
